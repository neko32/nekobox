## 2. 発見事項

### 🔴 Critical (修正済み)

#### C-01: バインドアドレスが全インターフェース (`0.0.0.0`)

| 項目 | 内容 |
|------|------|
| 場所 | `backend/src/main.rs` |
| 問題 | `0.0.0.0:8080` にバインドしており、同一 LAN 内の他デバイスから `POST /v1/msg` に到達可能だった |
| リスク | ローカルコンパニオンアプリが意図しない外部ホストに利用される可能性 |
| 対応 | 環境変数 `NEKOBOX_BIND_HOST` を新設し、デフォルトを `127.0.0.1` に変更。Docker コンテナ内では `NEKOBOX_BIND_HOST=0.0.0.0` を設定することで対応。 |
| 状態 | ✅ 修正済み (v0.1.0) |

---

### 🟡 Medium (受け入れ・ローカルアプリ限定リスク)

#### M-01: エラーレスポンスに内部パスが含まれる可能性

| 項目 | 内容 |
|------|------|
| 場所 | `backend/src/core/config.rs`, `backend/src/core/error.rs` |
| 問題 | `AppError::Config` の `Display` に `{path}: {e}` 形式でファイルパスが含まれ、HTTP 502/500 レスポンスの `message` フィールドに露出する |
| リスク | クライアント（Godot フロントエンド）にサーバのディレクトリ構造が漏洩する可能性 |
| 評価 | **ローカル専用アプリ**のため、クライアントは同一マシン上の Godot プロセスのみ。実質的な攻撃面はない。 |
| 推奨 | 将来ネットワーク越しに使用する場合は内部エラーを `INTERNAL_ERROR` に抽象化すること |
| 状態 | 🟡 受け入れ（ローカル限定リスク） |

#### M-02: メッセージ長の上限なし

| 項目 | 内容 |
|------|------|
| 場所 | `backend/src/api/routes/msg.rs` (`MsgRequest.message`) |
| 問題 | `message` フィールドにバイト数制限がなく、大きなペイロードを LM Studio へ転送できる |
| リスク | LM Studio のコンテキスト上限超過によるエラー、またはメモリ圧迫 |
| 評価 | ローカルアプリかつ LM Studio 側でコンテキストウィンドウ制限があるため実被害は限定的 |
| 推奨 | 将来的に `message` に最大 4096 バイト程度の制限を追加すること |
| 状態 | 🟡 受け入れ（将来の改善候補） |

---

### ✅ Safe (問題なし)

#### S-01: SQL インジェクション対策

- `sqlx::query().bind()` によるパラメータ化クエリを使用
- ユーザー入力は直接 SQL 文字列に埋め込まれていない ✓

#### S-02: パストラバーサル対策

- `load_system_prompt()` で使用するファイルパスは `AppConfig`（起動時にロード済み）から取得
- `req.character_name` / `req.version` はファイルパス生成に使用されていない ✓
- `settings_name` として DB に保存されるのみ ✓

#### S-03: LM Studio との通信

- `http://127.0.0.1:PORT` による localhost 通信は意図的設計
- LM Studio はローカルプロセスのため TLS は不要 ✓

#### S-04: 認証なし設計

- `/v1/msg` エンドポイントは認証なし
- ローカル専用アプリかつ `127.0.0.1` バインドのため、認証は不要と判断 ✓
- C-01 修正後、アクセスは同一マシン内のプロセスに限定される

#### S-05: 依存ライブラリの脆弱性

- `cargo audit` 実施：クリーン（既知の脆弱性ゼロ）
- RUSTSEC-2023-0071 は `sqlx-mysql` 経由の間接依存であり、本アプリは SQLite のみ使用のため実行時リスクなし（`.cargo/audit.toml` にて文書化・除外済み）

## 5. 総評

Phase 1 の実装は **ローカル専用デスクトップコンパニオン**として適切な設計を持つ。

Critical な問題であったバインドアドレスの修正後、残存リスクはいずれもローカルアプリの性質上許容できる範囲にある。SQL インジェクション対策・パストラバーサル対策は正しく実装されており、依存ライブラリに既知の悪用可能な脆弱性は存在しない。

Phase 2 でネットワーク越しの通信が追加される場合は、認証機能とエラー情報のサニタイズを最優先で実施すること。
